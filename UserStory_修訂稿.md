## 跨功能補充規格（1-9 新增）

### 冷卻時間與防濫用規則
1. 公告互動冷卻：同一使用者對同一公告的重複按讚/取消按讚操作需間隔 3 秒；留言連續送出間隔 10 秒。
2. 投票送出限制：同一帳號對同一投票僅允許一次有效送出；重複送出請求需間隔 30 秒並回傳已投票提示。
3. LINE 推播去重：相同事件（同公告/同投票/同包裹/同緊急事件）10 分鐘內不得重複推播給同一對象。
4. 費用查詢節流：同一住戶透過 LINE Bot 連續查詢管理費的最短間隔為 5 秒。
5. 緊急按鈕冷卻：同一 IoT 裝置緊急按鈕觸發後 30 秒內再次按壓視為同一事件續報，不重複開新單。
6. 報修防重送：同一住戶於 10 分鐘內提交「同地點 + 同類別 + 高相似描述」報修時，系統須提示疑似重複。
7. 包裹 IoT 聲響冷卻：同戶提醒聲最短間隔 60 秒，避免短時間重複觸發造成擾民。
8. 登入保護：同帳號連續 5 次登入失敗後鎖定 15 分鐘（冷卻時間），期間拒絕登入請求。
9. 設施預約冷卻：同一住戶取消同一設施時段後，5 分鐘內不得重新預約同時段。
10. 全域審計：所有觸發冷卻、節流、鎖定與去重的事件必須寫入審計紀錄（時間、帳號、來源、規則代碼）。
11. 訪客 IoT 聲響冷卻：同一住戶訪客到訪提醒音最短間隔 60 秒，避免重複觸發。

## 功能 1：公告（Epic）

### Epic 說明
- 角色：社區住戶、社區管理員
- 需求：在系統中發布、查閱、管理公告，並可即時通知住戶
- 價值：
  - 住戶可即時掌握停水、維修、活動等重要訊息
  - 管理端可統一發布與追蹤公告，降低資訊落差與遺漏

### 1.1 查看公告（優先順序：1）
**User Story**
作為社區住戶，我想在公告列表看到最新公告與基本資訊，以便我能快速掌握近期社區動態。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已登入並進入公告頁
   When 系統載入公告資料
   Then 顯示每筆公告的標題、發布日期、發布者與摘要。
2. Given 公告列表已載入
   When 顯示排序結果
   Then 公告應依發布時間由新到舊排序。
3. Given 系統目前無公告
   When 使用者進入公告頁
   Then 顯示「目前尚無公告」提示訊息。

### 1.2 查看公告詳情、按讚與留言（優先順序：1）
**User Story**
作為社區住戶，我想點擊公告查看完整內容，並可按讚與留言，以便我能互動與回饋。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者位於公告列表頁
   When 點擊任一公告
   Then 進入公告詳情頁並顯示完整內容、附件/圖片（若有）。
2. Given 使用者已進入公告詳情頁
   When 點擊按讚
   Then 系統更新按讚狀態與讚數。
3. Given 使用者已進入公告詳情頁
   When 輸入留言並送出
   Then 系統新增留言並顯示留言者、留言時間與內容。
4. Given 使用者短時間重複按讚或連續送出留言
   When 未達互動冷卻時間
   Then 系統阻止重複操作並提示稍後再試。

### 1.3 發布公告（優先順序：1）
**User Story**
作為管理員，我想在後台新增公告，以便我能即時發布社區重要訊息。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入公告管理頁
   When 填寫標題、內容、發布狀態（草稿/發布）與可選欄位（圖片、附件）後按下儲存
   Then 系統成功建立公告並回傳成功訊息。
2. Given 公告建立成功且狀態為發布
   When 返回公告列表
   Then 新公告應出現在列表前段（依時間排序）。
3. Given 管理員缺少必填欄位（如標題或內容）
   When 嘗試儲存
   Then 系統阻止送出並提示缺漏欄位。

### 1.4 修改與刪除公告（優先順序：1）
**User Story**
作為管理員，我想修改或刪除已發布公告，以便在資訊錯誤或過期時能及時處理。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員在公告管理頁選擇某公告
   When 編輯內容並儲存
   Then 系統更新公告並在列表中即時顯示最新內容。
2. Given 管理員選擇刪除公告
   When 於二次確認視窗按下確認
   Then 系統刪除該公告並自列表移除。
3. Given 公告已被其他人更新或刪除
   When 管理員嘗試提交舊資料
   Then 系統提示資料已變更，要求重新整理。

### 1.5 公告分類與搜尋（優先順序：2）
**User Story**
作為管理員，我想透過分類與關鍵字快速查找公告，以便我能有效維護大量公告資料。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員進入公告管理頁
   When 以關鍵字搜尋（標題、內容、作者）
   Then 列表顯示符合條件的公告。
2. Given 管理員設定篩選條件（狀態、分類、時間區間）
   When 套用篩選
   Then 只顯示符合條件的公告。
3. Given 管理員清除所有篩選條件
   When 返回列表
   Then 顯示完整公告資料集。

### 1.6 公告推播至 LINE Bot（優先順序：3）
**User Story**
作為管理員，我想在發布公告後自動推播 LINE Bot，以便住戶能即時接收重要通知。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員發布公告且公告狀態為「發布」
   When 公告儲存成功
   Then 系統自動觸發 LINE 推播流程。
2. Given 住戶已綁定 LINE Bot
   When 推播執行成功
   Then 住戶收到包含公告標題、摘要、時間與連結的訊息。
3. Given 推播服務失敗或部分住戶推送失敗
   When 系統完成推播嘗試
   Then 後台應保留錯誤紀錄供管理員追蹤與重送。

## 功能 2：投票（Epic）

### Epic 說明
- 角色：社區住戶、社區管理員
- 需求：建立、發布、參與並追蹤投票活動，並透過公告或 LINE Bot 接收通知
- 價值：
  - 住戶可參與社區決策並表達意見
  - 管理端可用一致流程管理投票週期，提高透明度與參與率

### 2.1 進入投票連結（優先順序：1）
**User Story**
作為社區住戶，我想透過 LINE Bot 或系統公告快速取得投票連結，以便我能即時進入投票頁並了解議題。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已建立並發布投票活動
   When 住戶查看 LINE Bot 通知或系統公告
   Then 系統提供可點擊的投票連結與投票摘要資訊（標題、截止時間）。
2. Given 住戶點擊投票連結
   When 系統導向外部投票頁
   Then 成功開啟正確投票表單，不出現失效或錯誤頁面。
3. Given 投票活動已截止或尚未開始
   When 住戶嘗試開啟連結
   Then 系統顯示對應狀態提示（未開始/已截止）。

### 2.2 參與投票（優先順序：1）
**User Story**
作為社區住戶，我想在投票表單提交我的選項，以便我能參與社區公共議題決策。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已進入有效投票表單
   When 填寫必填欄位並送出
   Then 系統完成提交並顯示成功訊息。
2. Given 住戶送出投票
   When 提交成功
   Then 投票資料應被記錄於指定資料來源（如 Google Sheet/投票資料表）。
3. Given 住戶重複送出同一活動且系統規則為一戶一票
   When 系統比對到重複身分
   Then 阻止重複提交並提示已投票。

### 2.3 接收投票結果（優先順序：3）
**User Story**
作為社區住戶，我想在投票截止後看到統計結果與決議公告，以便我能了解最終共識與執行方向。

**Acceptance Criteria（Given / When / Then）**
1. Given 某投票活動已截止
   When 管理員發布投票結果公告
   Then 住戶可在系統公告頁查看最終統計結果與決議內容。
2. Given 住戶已綁定 LINE Bot
   When 結果公告發布完成
   Then 住戶收到結果通知與公告連結。
3. Given 投票尚未截止
   When 住戶嘗試查看最終結果
   Then 系統顯示「尚未結算」或僅顯示目前進度（依管理規則）。

### 2.4 發布與管理投票（優先順序：2）
**User Story**
作為管理員，我想建立、編輯與下架投票活動，以便我能維護議題內容與投票時程。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入投票管理頁
   When 建立新投票並填寫標題、說明、選項、截止時間、狀態
   Then 系統儲存成功並顯示於投票列表。
2. Given 管理員選擇既有投票
   When 編輯內容後儲存
   Then 系統更新該投票資料並記錄更新時間。
3. Given 管理員刪除或下架投票
   When 確認操作
   Then 投票應從住戶可參與清單移除，且保留後台操作紀錄。

### 2.5 投票推播至 LINE Bot（優先順序：3）
**User Story**
作為社區住戶，我想在有新投票時收到 LINE Bot 通知，以便我不會錯過重要表決。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員發布新投票活動
   When 發布成功
   Then 系統自動觸發 LINE 推播，內容包含標題、截止時間與投票連結。
2. Given 住戶已綁定 LINE Bot
   When 推播發送完成
   Then 住戶收到投票通知訊息。
3. Given 推播過程出現失敗
   When 系統完成發送流程
   Then 後台應記錄失敗原因，供管理員重送或追蹤。
4. Given 相同投票事件已於短時間內送達
   When 系統再次觸發推播
   Then 套用推播去重冷卻，避免重複通知。

## 功能 3：費用管理（Epic）

### Epic 說明
- 角色：社區住戶、社區管理員
- 需求：管理與查詢管理費、社區支出、財務報表與 LINE Bot 費用通知
- 價值：
  - 住戶可清楚掌握個人繳費狀態與社區費用流向
  - 管理端可降低人工帳務錯誤並提升財務透明度

### 3.1 查看管理費與社區支出（優先順序：1）
**User Story**
作為社區住戶，我想查看每月管理費繳費狀態與社區支出明細，以便我能確認繳費進度並了解社區資金使用狀況。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已登入並進入費用頁（收入/支出）
   When 系統載入個人管理費與社區支出資料
   Then 顯示每月金額、繳費狀態、到期日與支出明細。
2. Given 住戶切換月份或年度
   When 套用時間條件
   Then 列表更新為對應期間資料。
3. Given 查詢期間無資料
   When 住戶進入頁面
   Then 顯示無資料提示而非錯誤訊息。

### 3.2 管理者管理管理費紀錄（優先順序：2）
**User Story**
作為管理員，我想新增並查看住戶管理費紀錄，以便我能維持完整且正確的收費資料。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入管理費收入頁
   When 新增住戶管理費紀錄（住戶、月份、金額、到期日、狀態）
   Then 系統成功儲存並在列表顯示新紀錄。
2. Given 管理員開啟收入列表
   When 依條件檢視或篩選
   Then 系統顯示對應住戶與月份的繳費資訊。
3. Given 必填欄位未完成
   When 管理員嘗試儲存
   Then 系統阻止送出並提示缺漏欄位。

### 3.3 管理者修改或刪除管理費紀錄（優先順序：2）
**User Story**
作為管理員，我想修改或刪除管理費紀錄，以便我能修正錯誤資料並維持帳務正確性。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員於收入列表選擇一筆紀錄
   When 修改欄位後儲存
   Then 系統更新該筆紀錄並顯示成功訊息。
2. Given 管理員於收入列表選擇刪除
   When 二次確認刪除
   Then 系統移除該筆紀錄並即時更新列表。
3. Given 紀錄已被他人異動
   When 管理員提交舊版資料
   Then 系統提示資料已更新並要求重新整理。

### 3.4 管理者管理社區支出紀錄（優先順序：2）
**User Story**
作為管理員，我想新增並查看社區支出紀錄，以便我能完整掌握社區費用用途與支出流向。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入管理費支出頁
   When 新增支出紀錄（類別、金額、日期、用途、備註）
   Then 系統成功儲存並顯示於支出列表。
2. Given 管理員進入支出列表
   When 依月份或支出類別篩選
   Then 系統顯示符合條件的支出資料。
3. Given 缺少必填資料
   When 嘗試送出
   Then 系統阻止儲存並顯示欄位提示。

### 3.5 管理者修改或刪除社區支出紀錄（優先順序：2）
**User Story**
作為管理員，我想修改或刪除支出紀錄，以便我能修正錯誤並保持財務資料一致。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已在支出列表選擇目標紀錄
   When 編輯資料後儲存
   Then 系統更新該紀錄並顯示成功訊息。
2. Given 管理員選擇刪除支出紀錄
   When 確認刪除
   Then 系統移除該筆支出並更新列表。
3. Given 刪除或更新操作失敗
   When 系統回傳錯誤
   Then 顯示明確原因並保留原始資料狀態。

### 3.6 查看財務報表（優先順序：2）
**User Story**
作為社區住戶與管理員，我想查看每月收支報表與損益分析，以便我能了解社區整體財務健康度。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已登入並進入財務報表頁
   When 選擇年份與月份
   Then 系統顯示總收入、總支出與本期損益。
2. Given 報表資料已計算完成
   When 顯示分析圖表
   Then 系統呈現支出類別占比與趨勢資訊。
3. Given 查詢區間資料不足
   When 系統產生報表
   Then 仍顯示可用統計並標示資料缺口。

### 3.7 費用管理推播至 LINE Bot（優先順序：3）
**User Story**
作為社區住戶，我想透過 LINE Bot 查詢繳費狀態並接收繳費提醒，以便我能即時掌握應繳費用並避免逾期。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已完成 LINE 帳號綁定
   When 在 LINE Bot 發送費用查詢指令
   Then 系統回覆最新管理費金額、狀態、到期日與房號資訊。
2. Given 系統偵測到即將到期或逾期帳單
   When 觸發通知排程
   Then LINE Bot 自動推播繳費提醒給對應住戶。
3. Given LINE 推播失敗
   When 系統完成通知流程
   Then 後台記錄失敗原因並提供重送追蹤資訊。
4. Given 使用者連續觸發費用查詢
   When 請求間隔小於查詢冷卻時間
   Then 系統暫緩處理並提示查詢過於頻繁。

## 功能 5：緊急通知（Epic）

### Epic 說明
- 角色：社區住戶、社區管理員、緊急聯絡人
- 需求：透過 IoT 裝置快速通報緊急事件，並同步通知管理端與聯絡人，完成後續處理追蹤
- 價值：
  - 協助長者或住戶在突發狀況下快速求援
  - 建立可追蹤的事件處理流程，提升社區安全與應變效率

### 5.1 住戶透過 IoT 裝置發出緊急通知（優先順序：1）
**User Story**
作為年長住戶，我想透過 IoT 裝置一鍵發出求救訊號，以便我在危急時能立即獲得協助。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已完成 IoT 裝置綁定且裝置連線正常
   When 住戶按下緊急按鈕
   Then 系統立即建立一筆緊急事件，並記錄觸發時間與裝置資訊。
2. Given 緊急事件已建立
   When 系統完成事件寫入
   Then 事件預設狀態為「未處理」。
3. Given 裝置離線或訊號中斷
   When 住戶按下緊急按鈕
   Then 系統於恢復連線後補送事件，並保留離線紀錄。
4. Given 同一裝置在短時間內重複按壓緊急按鈕

   When 核對身分後點擊訪客簽到
   Then 該筆紀錄狀態更新為 Checked In。
2. Given 簽到成功
   When 系統刷新列表
   Then 該筆紀錄移動至訪客中區塊並記錄簽到時間。
3. Given 訪客已簽到
   When 再次執行簽到

### 10.4 警衛執行訪客簽退（優先順序：1）

   When 點擊訪客簽退
   Then 該筆紀錄狀態更新為 Checked Out。
2. Given 簽退成功
   When 系統刷新列表
   Then 該筆紀錄移動至訪客歷史區塊並記錄簽退時間。
3. Given 訪客尚未簽到
   When 嘗試直接簽退
   Then 系統阻止操作並提示流程順序錯誤。

### 10.5 訪客紀錄搜尋與篩選（優先順序：3）
**User Story**
作為警衛與社區住戶，我想用關鍵字搜尋訪客紀錄，以便快速查找來訪歷史與聯絡資訊。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已登入訪客頁

2. Given 使用者套用狀態或日期篩選
   When 套用條件
   Then 僅顯示符合條件的訪客紀錄。
3. Given 查無符合資料
   When 搜尋完成
   Then 顯示無符合結果提示。

### 10.6 IoT 提醒訪客上門（優先順序：3）
**User Story**
作為社區住戶，我想在訪客到訪時由 IoT 裝置發出提示音，以便我不登入系統也能即時得知訪客到達。

   Then 後台標記失敗並提供重送機制。

   Then 裝置不重複播放提示音，僅更新通知紀錄。

## 功能 11：會議紀錄（Epic）

### Epic 說明
- 角色：社區住戶、社區管理員（管委會）
- 需求：建立數位化的會議紀錄管理系統，管理員可發布會議內容與電子檔，住戶可查閱與下載
- 價值：
  - 資訊透明化：確保住戶了解社區決策過程與結果，減少資訊落差
  - 無紙化保存：取代傳統紙本公告，方便隨時調閱歷史紀錄，不佔實體空間
  - 完整性留存：除系統摘要外，提供正式簽核的 PDF 檔下載，確保文件具備法律效力

### 11.1 住戶瀏覽與搜尋會議列表（優先順序：2）
**User Story**
作為社區住戶，我想瀏覽會議紀錄列表並透過關鍵字搜尋特定會議，以便我能快速找到感興趣的會議場次，了解社區動態。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已登入並進入會議紀錄頁面
   When 系統載入資料
   Then 顯示會議卡片列表，包含主題、日期、地點，若有附件則顯示「附有完整會議紀錄」圖示。
2. Given 住戶已進入會議列表頁
   When 在搜尋框輸入關鍵字（如會議主題、地點、日期）
   Then 列表即時篩選並顯示符合條件的會議卡片。
3. Given 查詢條件無符合結果
   When 搜尋完成
   Then 系統顯示無符合會議提示。

### 11.2 查看會議詳情與下載檔案（優先順序：1）
**User Story**
作為社區住戶，我想點擊會議查看詳細內容並下載完整 PDF 檔，以便我能深入了解具體決議事項並保存正式文件備查。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已登入並進入會議列表
   When 點擊某場會議卡片
   Then 進入詳情頁面，顯示會議標題、時間、地點、出席人員與重點摘要。
2. Given 住戶已進入會議詳情頁面
   When 點擊「下載完整會議紀錄」按鈕
   Then 系統提供 PDF 檔案下載（若有上傳）。
3. Given 會議紀錄未上傳 PDF 附件
   When 住戶進入詳情頁面
   Then 系統仍顯示重點摘要，但下載按鈕呈灰顯狀態。

### 11.3 管理員發布與維護會議紀錄（優先順序：1）
**User Story**
作為社區管理員，我想新增、編輯或刪除會議紀錄，包含輸入重點摘要與上傳 PDF，以便我能將紙本會議轉為數位檔案發布給住戶。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入會議管理頁
   When 點擊新增會議並填寫主題、時間、地點、人員名單與重點摘要，並上傳 PDF 後儲存

2. Given 管理員選擇既有會議

   Then 系統更新該筆紀錄，住戶端反映最新內容。
3. Given 管理員在管理列表選擇刪除
   When 於二次確認視窗按下確認
   Then 系統刪除該會議紀錄並自列表移除。

   Then 顯示表格包含姓名、房號、電話、Email、身分（住戶/管委會/警衛）與關係（戶主/住戶成員/租客）。
2. Given 住戶列表已載入
   When 顯示排序結果
   Then 列表依房號由小到大排序，同房號依姓名排序。
3. Given 系統目前無住戶資料
   When 管理員進入頁面
   Then 顯示「目前尚無住戶資料」提示訊息。

### 12.2 管理員新增住戶資料（優先順序：1）
**User Story**
作為社區管理員，我想新增搬入的住戶或新進工作人員的資料，以便系統能識別該使用者並讓其開始使用社區系統服務。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入住戶/人員管理頁面
   When 點擊「新增一筆」按鈕並填寫姓名、房號、電話、Email，選擇身分（Role）與關係（Relationship）後點擊「新增」
   Then 系統儲存該筆資料，列表即時顯示新住戶並關閉視窗。
2. Given 管理員成功新增住戶
   When 確認列表
   Then 新住戶出現於對應房號位置且身分與關係標籤正確顯示。
3. Given 必填欄位（姓名、房號）未填寫或 Email 格式錯誤
   When 管理員嘗試送出
   Then 系統阻止儲存並提示需修正的欄位。

### 12.3 管理員編輯住戶資料（優先順序：2）
**User Story**
作為社區管理員，我想修改住戶的聯絡方式或角色權限，以便我能維護資料的正確性（例如：電話變更、管委會改選或租客轉為戶主）。

**Acceptance Criteria（Given / When / Then）**

   When 點擊「編輯」圖示，在彈出視窗中修改電話、Email 或身分權限後點擊「儲存變更」

2. Given 管理員僅更新電話或身分
   When 送出儲存
   Then 系統僅更新對應欄位，其餘欄位保持不變。
3. Given 資料已被他人異動
   When 管理員嘗試提交舊版資料
   Then 系統提示資料已變更並要求重新整理。

### 12.4 管理員刪除住戶資料（優先順序：3）
**User Story**
作為社區管理員，我想刪除已遷出住戶或離職人員的資料，以便資料庫不會累積過期無效資料並確保系統安全性。

**Acceptance Criteria（Given / When / Then）**

   When 點擊「刪除」圖示並於確認視窗確認操作
   Then 該筆住戶資料從系統移除，不再顯示於列表。

   When 在確認視窗取消
   Then 系統不執行刪除並保留原資料。
3. Given 刪除操作失敗（如資料被其他功能引用）
   When 系統回傳錯誤
   Then 顯示失敗原因（如「住戶有未完成訪客/報修紀錄」）並阻止刪除。

### 12.5 搜尋住戶資料（優先順序：2）
**User Story**
作為社區管理員，我想透過關鍵字快速找到特定住戶，以便我能在緊急狀況或需要聯繫特定房號時迅速找到對象。

**Acceptance Criteria（Given / When / Then）**

   Then 列表自動篩選並僅顯示符合條件的住戶資料。
2. Given 管理員套用身分或關係篩選條件
   When 選擇篩選值
   Then 僅顯示符合該身分或關係的住戶。
3. Given 查詢條件無符合結果
   When 搜尋完成
   Then 顯示無符合住戶提示而非錯誤訊息。

   When 以日期、人員或狀態篩選
   Then 系統顯示符合條件的交接案件。
2. Given 管理員查看單筆案件
   When 展開詳情
   Then 可看到完整交接項目、簽核時間與異動紀錄。
3. Given 交接案件含附件
   When 開啟案件詳情
   Then 可下載或預覽附件檔案。

## 功能 22：資料解密申請與審核（Epic）

### Epic 說明
- 角色：社區管理員、授權審核人員
- 需求：對敏感資料建立解密申請、審核與留痕流程
- 價值：
  - 降低敏感資料濫用風險
  - 建立最小權限與雙重確認機制
  - 符合資安與稽核要求

### 22.1 提交解密申請（優先順序：1）
**User Story**
作為社區管理員，我想在需要時提交解密申請，以便在合法授權下檢視敏感資料。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者具備解密申請權限
   When 填寫申請原因、資料範圍、有效期限後送出
   Then 系統建立解密申請單並標記為待審核。
2. Given 申請建立成功
   When 系統通知審核人
   Then 審核人收到待審通知與申請摘要。
3. Given 使用者短時間重複提交同類申請
   When 未達申請冷卻間隔
   Then 系統阻止重複送出並提示稍後再試。

### 22.2 解密審核流程（優先順序：1）
**User Story**
作為授權審核人員，我想審核解密申請，以便只允許符合規範的資料存取。

**Acceptance Criteria（Given / When / Then）**
1. Given 審核人已進入審核清單
   When 開啟待審申請
   Then 顯示申請原因、申請人、範圍與有效期限。
2. Given 審核人判定申請合理
   When 核准申請
   Then 系統建立限時解密授權並記錄審核資訊。
3. Given 審核人判定申請不合理
   When 駁回申請
   Then 系統通知申請人駁回原因並結束流程。

### 22.3 解密執行與留痕（優先順序：1）
**User Story**
作為稽核人員，我想追蹤每次解密操作紀錄，以便事後可完整還原敏感資料存取過程。

**Acceptance Criteria（Given / When / Then）**
1. Given 申請已核准且仍在授權有效期內
   When 使用者執行解密查詢
   Then 系統回傳解密結果並寫入操作留痕。
2. Given 授權已過期或被撤銷
   When 使用者嘗試解密
   Then 系統拒絕請求並提示授權失效。
3. Given 解密操作完成
   When 稽核人員查詢紀錄
   Then 可看到申請單號、操作時間、資料範圍與操作者資訊。
