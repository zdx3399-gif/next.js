## 功能 1：公告（Epic）

### Epic 說明
- 角色：社區住戶、社區管理員
- 需求：在系統中發布、查閱、管理公告，並可即時通知住戶
- 價值：
  - 住戶可即時掌握停水、維修、活動等重要訊息
  - 管理端可統一發布與追蹤公告，降低資訊落差與遺漏

### 1.1 查看公告（優先順序：1）
**User Story**
作為社區住戶，我想在公告列表看到最新公告與基本資訊，以便我能快速掌握近期社區動態。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已登入並進入公告頁
   When 系統載入公告資料
   Then 顯示每筆公告的標題、發布日期、發布者與摘要。
2. Given 公告列表已載入
   When 顯示排序結果
   Then 公告應依發布時間由新到舊排序。
3. Given 系統目前無公告
   When 使用者進入公告頁
   Then 顯示「目前尚無公告」提示訊息。

### 1.2 查看公告詳情、按讚與留言（優先順序：1）
**User Story**
作為社區住戶，我想點擊公告查看完整內容，並可按讚與留言，以便我能互動與回饋。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者位於公告列表頁
   When 點擊任一公告
   Then 進入公告詳情頁並顯示完整內容、附件/圖片（若有）。
2. Given 使用者已進入公告詳情頁
   When 點擊按讚
   Then 系統更新按讚狀態與讚數。
3. Given 使用者已進入公告詳情頁
   When 輸入留言並送出
   Then 系統新增留言並顯示留言者、留言時間與內容。

### 1.3 發布公告（優先順序：1）
**User Story**
作為管理員，我想在後台新增公告，以便我能即時發布社區重要訊息。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入公告管理頁
   When 填寫標題、內容、發布狀態（草稿/發布）與可選欄位（圖片、附件）後按下儲存
   Then 系統成功建立公告並回傳成功訊息。
2. Given 公告建立成功且狀態為發布
   When 返回公告列表
   Then 新公告應出現在列表前段（依時間排序）。
3. Given 管理員缺少必填欄位（如標題或內容）
   When 嘗試儲存
   Then 系統阻止送出並提示缺漏欄位。

### 1.4 修改與刪除公告（優先順序：1）
**User Story**
作為管理員，我想修改或刪除已發布公告，以便在資訊錯誤或過期時能及時處理。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員在公告管理頁選擇某公告
   When 編輯內容並儲存
   Then 系統更新公告並在列表中即時顯示最新內容。
2. Given 管理員選擇刪除公告
   When 於二次確認視窗按下確認
   Then 系統刪除該公告並自列表移除。
3. Given 公告已被其他人更新或刪除
   When 管理員嘗試提交舊資料
   Then 系統提示資料已變更，要求重新整理。

### 1.5 公告分類與搜尋（優先順序：2）
**User Story**
作為管理員，我想透過分類與關鍵字快速查找公告，以便我能有效維護大量公告資料。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員進入公告管理頁
   When 以關鍵字搜尋（標題、內容、作者）
   Then 列表顯示符合條件的公告。
2. Given 管理員設定篩選條件（狀態、分類、時間區間）
   When 套用篩選
   Then 只顯示符合條件的公告。
3. Given 管理員清除所有篩選條件
   When 返回列表
   Then 顯示完整公告資料集。

### 1.6 公告推播至 LINE Bot（優先順序：3）
**User Story**
作為管理員，我想在發布公告後自動推播 LINE Bot，以便住戶能即時接收重要通知。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員發布公告且公告狀態為「發布」
   When 公告儲存成功
   Then 系統自動觸發 LINE 推播流程。
2. Given 住戶已綁定 LINE Bot
   When 推播執行成功
   Then 住戶收到包含公告標題、摘要、時間與連結的訊息。
3. Given 推播服務失敗或部分住戶推送失敗
   When 系統完成推播嘗試
   Then 後台應保留錯誤紀錄供管理員追蹤與重送。

## 功能 2：投票（Epic）

### Epic 說明
- 角色：社區住戶、社區管理員
- 需求：建立、發布、參與並追蹤投票活動，並透過公告或 LINE Bot 接收通知
- 價值：
  - 住戶可參與社區決策並表達意見
  - 管理端可用一致流程管理投票週期，提高透明度與參與率

### 2.1 進入投票連結（優先順序：1）
**User Story**
作為社區住戶，我想透過 LINE Bot 或系統公告快速取得投票連結，以便我能即時進入投票頁並了解議題。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已建立並發布投票活動
   When 住戶查看 LINE Bot 通知或系統公告
   Then 系統提供可點擊的投票連結與投票摘要資訊（標題、截止時間）。
2. Given 住戶點擊投票連結
   When 系統導向外部投票頁
   Then 成功開啟正確投票表單，不出現失效或錯誤頁面。
3. Given 投票活動已截止或尚未開始
   When 住戶嘗試開啟連結
   Then 系統顯示對應狀態提示（未開始/已截止）。

### 2.2 參與投票（優先順序：1）
**User Story**
作為社區住戶，我想在投票表單提交我的選項，以便我能參與社區公共議題決策。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已進入有效投票表單
   When 填寫必填欄位並送出
   Then 系統完成提交並顯示成功訊息。
2. Given 住戶送出投票
   When 提交成功
   Then 投票資料應被記錄於指定資料來源（如 Google Sheet/投票資料表）。
3. Given 住戶重複送出同一活動且系統規則為一戶一票
   When 系統比對到重複身分
   Then 阻止重複提交並提示已投票。

### 2.3 接收投票結果（優先順序：3）
**User Story**
作為社區住戶，我想在投票截止後看到統計結果與決議公告，以便我能了解最終共識與執行方向。

**Acceptance Criteria（Given / When / Then）**
1. Given 某投票活動已截止
   When 管理員發布投票結果公告
   Then 住戶可在系統公告頁查看最終統計結果與決議內容。
2. Given 住戶已綁定 LINE Bot
   When 結果公告發布完成
   Then 住戶收到結果通知與公告連結。
3. Given 投票尚未截止
   When 住戶嘗試查看最終結果
   Then 系統顯示「尚未結算」或僅顯示目前進度（依管理規則）。

### 2.4 發布與管理投票（優先順序：2）
**User Story**
作為管理員，我想建立、編輯與下架投票活動，以便我能維護議題內容與投票時程。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入投票管理頁
   When 建立新投票並填寫標題、說明、選項、截止時間、狀態
   Then 系統儲存成功並顯示於投票列表。
2. Given 管理員選擇既有投票
   When 編輯內容後儲存
   Then 系統更新該投票資料並記錄更新時間。
3. Given 管理員刪除或下架投票
   When 確認操作
   Then 投票應從住戶可參與清單移除，且保留後台操作紀錄。

### 2.5 投票推播至 LINE Bot（優先順序：3）
**User Story**
作為社區住戶，我想在有新投票時收到 LINE Bot 通知，以便我不會錯過重要表決。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員發布新投票活動
   When 發布成功
   Then 系統自動觸發 LINE 推播，內容包含標題、截止時間與投票連結。
2. Given 住戶已綁定 LINE Bot
   When 推播發送完成
   Then 住戶收到投票通知訊息。
3. Given 推播過程出現失敗
   When 系統完成發送流程
   Then 後台應記錄失敗原因，供管理員重送或追蹤。

## 功能 3：費用管理（Epic）

### Epic 說明
- 角色：社區住戶、社區管理員
- 需求：管理與查詢管理費、社區支出、財務報表與 LINE Bot 費用通知
- 價值：
  - 住戶可清楚掌握個人繳費狀態與社區費用流向
  - 管理端可降低人工帳務錯誤並提升財務透明度

### 3.1 查看管理費與社區支出（優先順序：1）
**User Story**
作為社區住戶，我想查看每月管理費繳費狀態與社區支出明細，以便我能確認繳費進度並了解社區資金使用狀況。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已登入並進入費用頁（收入/支出）
   When 系統載入個人管理費與社區支出資料
   Then 顯示每月金額、繳費狀態、到期日與支出明細。
2. Given 住戶切換月份或年度
   When 套用時間條件
   Then 列表更新為對應期間資料。
3. Given 查詢期間無資料
   When 住戶進入頁面
   Then 顯示無資料提示而非錯誤訊息。

### 3.2 管理者管理管理費紀錄（優先順序：2）
**User Story**
作為管理員，我想新增並查看住戶管理費紀錄，以便我能維持完整且正確的收費資料。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入管理費收入頁
   When 新增住戶管理費紀錄（住戶、月份、金額、到期日、狀態）
   Then 系統成功儲存並在列表顯示新紀錄。
2. Given 管理員開啟收入列表
   When 依條件檢視或篩選
   Then 系統顯示對應住戶與月份的繳費資訊。
3. Given 必填欄位未完成
   When 管理員嘗試儲存
   Then 系統阻止送出並提示缺漏欄位。

### 3.3 管理者修改或刪除管理費紀錄（優先順序：2）
**User Story**
作為管理員，我想修改或刪除管理費紀錄，以便我能修正錯誤資料並維持帳務正確性。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員於收入列表選擇一筆紀錄
   When 修改欄位後儲存
   Then 系統更新該筆紀錄並顯示成功訊息。
2. Given 管理員於收入列表選擇刪除
   When 二次確認刪除
   Then 系統移除該筆紀錄並即時更新列表。
3. Given 紀錄已被他人異動
   When 管理員提交舊版資料
   Then 系統提示資料已更新並要求重新整理。

### 3.4 管理者管理社區支出紀錄（優先順序：2）
**User Story**
作為管理員，我想新增並查看社區支出紀錄，以便我能完整掌握社區費用用途與支出流向。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已登入並進入管理費支出頁
   When 新增支出紀錄（類別、金額、日期、用途、備註）
   Then 系統成功儲存並顯示於支出列表。
2. Given 管理員進入支出列表
   When 依月份或支出類別篩選
   Then 系統顯示符合條件的支出資料。
3. Given 缺少必填資料
   When 嘗試送出
   Then 系統阻止儲存並顯示欄位提示。

### 3.5 管理者修改或刪除社區支出紀錄（優先順序：2）
**User Story**
作為管理員，我想修改或刪除支出紀錄，以便我能修正錯誤並保持財務資料一致。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已在支出列表選擇目標紀錄
   When 編輯資料後儲存
   Then 系統更新該紀錄並顯示成功訊息。
2. Given 管理員選擇刪除支出紀錄
   When 確認刪除
   Then 系統移除該筆支出並更新列表。
3. Given 刪除或更新操作失敗
   When 系統回傳錯誤
   Then 顯示明確原因並保留原始資料狀態。

### 3.6 查看財務報表（優先順序：2）
**User Story**
作為社區住戶與管理員，我想查看每月收支報表與損益分析，以便我能了解社區整體財務健康度。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已登入並進入財務報表頁
   When 選擇年份與月份
   Then 系統顯示總收入、總支出與本期損益。
2. Given 報表資料已計算完成
   When 顯示分析圖表
   Then 系統呈現支出類別占比與趨勢資訊。
3. Given 查詢區間資料不足
   When 系統產生報表
   Then 仍顯示可用統計並標示資料缺口。

### 3.7 費用管理推播至 LINE Bot（優先順序：3）
**User Story**
作為社區住戶，我想透過 LINE Bot 查詢繳費狀態並接收繳費提醒，以便我能即時掌握應繳費用並避免逾期。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已完成 LINE 帳號綁定
   When 在 LINE Bot 發送費用查詢指令
   Then 系統回覆最新管理費金額、狀態、到期日與房號資訊。
2. Given 系統偵測到即將到期或逾期帳單
   When 觸發通知排程
   Then LINE Bot 自動推播繳費提醒給對應住戶。
3. Given LINE 推播失敗
   When 系統完成通知流程
   Then 後台記錄失敗原因並提供重送追蹤資訊。

## 功能 5：緊急通知（Epic）

### Epic 說明
- 角色：社區住戶、社區管理員、緊急聯絡人
- 需求：透過 IoT 裝置快速通報緊急事件，並同步通知管理端與聯絡人，完成後續處理追蹤
- 價值：
  - 協助長者或住戶在突發狀況下快速求援
  - 建立可追蹤的事件處理流程，提升社區安全與應變效率

### 5.1 住戶透過 IoT 裝置發出緊急通知（優先順序：1）
**User Story**
作為年長住戶，我想透過 IoT 裝置一鍵發出求救訊號，以便我在危急時能立即獲得協助。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已完成 IoT 裝置綁定且裝置連線正常
   When 住戶按下緊急按鈕
   Then 系統立即建立一筆緊急事件，並記錄觸發時間與裝置資訊。
2. Given 緊急事件已建立
   When 系統完成事件寫入
   Then 事件預設狀態為「未處理」。
3. Given 裝置離線或訊號中斷
   When 住戶按下緊急按鈕
   Then 系統於恢復連線後補送事件，並保留離線紀錄。

### 5.2 系統通知警衛/管理端（優先順序：2）
**User Story**
作為社區管理者，我想在接收到緊急事件時即時收到通知，以便我能第一時間派員協助。

**Acceptance Criteria（Given / When / Then）**
1. Given 系統接收到住戶緊急事件
   When 事件建立成功
   Then 管理端與警衛端立即收到事件通知。
2. Given 管理端收到通知
   When 開啟事件詳情
   Then 顯示住戶資訊、位置、事件時間與來源裝置。
3. Given 通知傳送失敗
   When 系統完成通知流程
   Then 後台應記錄失敗原因並可重送。

### 5.3 系統通知緊急聯絡人（優先順序：2）
**User Story**
作為住戶的緊急聯絡人，我想在住戶發出求救時立即收到通知，以便我能盡快聯繫或到場協助。

**Acceptance Criteria（Given / When / Then）**
1. Given 住戶已設定至少一位緊急聯絡人
   When 系統接收到該住戶的緊急事件
   Then 系統通知所有有效聯絡人。
2. Given 聯絡人收到通知
   When 檢視通知內容
   Then 應包含住戶姓名、事件時間、位置與回撥方式。
3. Given 聯絡人資料不完整或失效
   When 系統嘗試發送
   Then 記錄失敗對象並通知管理端補正資料。

### 5.4 管理者處理緊急事件與紀錄（優先順序：2）
**User Story**
作為社區管理者，我想查看並更新緊急事件處理進度，以便我能追蹤處理過程並保留完整紀錄。

**Acceptance Criteria（Given / When / Then）**
1. Given 管理者已登入並進入緊急事件頁
   When 系統載入事件清單
   Then 顯示狀態（未處理/處理中/已完成）、住戶資訊與事件時間。
2. Given 管理者開啟某事件
   When 更新處理狀態與處理備註
   Then 系統儲存變更並更新時間軸紀錄。
3. Given 事件已標記完成
   When 管理者查詢歷史
   Then 可看到完整處理過程與負責人紀錄。

### 5.5 緊急事件 LINE Bot 推播（優先順序：2）
**User Story**
作為住戶的緊急聯絡人，我想在住戶觸發緊急事件時收到 LINE Bot 推播，以便我不用登入系統也能即時得知狀況。

**Acceptance Criteria（Given / When / Then）**

   When 住戶按下緊急按鈕
   Then 系統推播緊急通知至對應聯絡人的 LINE Bot。
2. Given 推播成功

**User Story**

   Then 系統成功建立報修紀錄並顯示提交成功訊息。
2. Given 住戶有現場照片可提供
   When 上傳照片後送出

### 6.2 住戶查看報修紀錄與進度（優先順序：2）
**User Story**

**User Story**

   When 系統載入報修清單

2. Given 管理員在搜尋框輸入關鍵字
   When 套用搜尋條件
   Then 列表顯示符合條件的報修案件。
3. Given 管理員切換篩選條件（如狀態）
   When 套用篩選

2. Given 案件狀態變更為已完成
   When 儲存成功
   Then 系統記錄完成時間與最終處理結果。
3. Given 儲存失敗
   When 系統回傳錯誤
   Then 顯示錯誤原因並保留原本資料狀態。

### 6.5 管理員刪除報修紀錄（優先順序：3）
**User Story**

**Acceptance Criteria（Given / When / Then）**
1. Given 管理員已在報修管理列表找到欲刪除項目
   When 點擊刪除並確認
   Then 系統移除該筆紀錄且不再顯示於列表。
2. Given 管理員誤觸刪除
   When 在確認視窗取消
   Then 系統不執行刪除並保留原資料。
3. Given 刪除操作失敗
   When 系統回傳結果
   Then 顯示失敗原因並保留重試入口。

## 功能 8：個人化（Epic）

### Epic 說明
- 角色：系統使用者（住戶、管委會、警衛）、管理員
- 需求：提供個人資料管理、登入登出、註冊與主題切換，建立個人化使用體驗
- 價值：
  - 使用者可確認身分並維護個資與帳號安全
  - 系統可依角色提供對應功能，提高操作效率與辨識度

### 8.1 使用者資料顯示（優先順序：1）
**User Story**
作為系統使用者，我想在介面上快速看到我的頭像、姓名與角色，以便我能確認目前登入帳號是否正確。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已登入
   When 進入系統任一主要頁面
   Then 左上角顯示頭像、姓名與身分角色。
2. Given 使用者資料已更新
   When 重新整理頁面
   Then 顯示最新姓名與角色資訊。
3. Given 使用者尚未設定頭像
   When 顯示個人區塊
   Then 系統顯示預設頭像並保留名稱與角色。

### 8.2 頭像上傳與更換（優先順序：3）
**User Story**
作為系統使用者，我想上傳或更換個人頭像，以便提升帳號辨識度與個人化體驗。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已登入
   When 在個人頭像區選擇圖片並上傳
   Then 系統顯示即時預覽並更新頭像。
2. Given 使用者上傳檔案格式或大小不符規範
   When 嘗試送出
   Then 系統阻止上傳並提示可接受規格。
3. Given 使用者成功更換頭像
   When 返回任一頁面
   Then 顯示新頭像且保存於個人設定。

### 8.3 修改個人資料與密碼（優先順序：1）
**User Story**
作為系統使用者，我想修改聯絡資訊與登入密碼，以便保持資訊即時且提高帳號安全。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已登入並開啟個人設定
   When 更新電話、Email 或密碼後儲存
   Then 系統驗證格式並成功更新資料。
2. Given 使用者僅更新電話或 Email
   When 送出儲存
   Then 系統僅更新對應欄位，不變更密碼。
3. Given 使用者修改密碼成功
   When 下次重新登入
   Then 僅新密碼可登入，舊密碼失效。

### 8.4 登入（優先順序：1）
**User Story**
作為系統使用者，我想使用註冊後帳號登入，以便我能進入對應角色功能頁。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已完成註冊
   When 輸入正確帳號密碼並提交
   Then 系統驗證成功並導向角色首頁。
2. Given 登入成功
   When 進入首頁
   Then 畫面顯示登入者名稱、角色與頭像。
3. Given 帳號或密碼錯誤
   When 使用者嘗試登入
   Then 系統顯示錯誤提示且不建立登入狀態。

### 8.5 登出（優先順序：1）
**User Story**
作為系統使用者，我想在使用完系統後安全登出，以便避免他人存取我的帳號資訊。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者已登入
   When 點擊登出按鈕
   Then 系統清除登入狀態並導回登入頁。
2. Given 使用者已登出
   When 嘗試返回受保護頁面
   Then 系統要求重新登入。
3. Given 登出流程執行完成
   When 檢查快取與權限資訊
   Then 本機暫存的敏感登入資訊已清除。

### 8.6 註冊（優先順序：1）
**User Story**
作為新住戶或新使用者，我想線上註冊帳號，以便我能開始使用社區系統功能。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者尚未擁有帳號
   When 填寫社區、Email、密碼、姓名、電話、單位與角色後送出
   Then 系統建立帳號並顯示註冊成功訊息。
2. Given 註冊成功
   When 流程結束
   Then 系統導向登入頁供使用者登入。
3. Given Email 已被註冊或資料格式不正確
   When 使用者送出註冊
   Then 系統顯示可理解錯誤訊息並阻止建立帳號。

### 8.7 切換系統主題模式（優先順序：1）
**User Story**
作為系統使用者，我想切換深色/淺色或自訂主題，以便依環境與偏好調整閱讀舒適度。

**Acceptance Criteria（Given / When / Then）**
1. Given 使用者位於系統任一頁面
   When 點擊主題切換按鈕
   Then 系統立即切換為對應主題樣式。
2. Given 使用者選擇自訂顏色主題
   When 套用設定
   Then 背景、文字與元件樣式同步更新。
3. Given 使用者已切換主題
   When 重新整理或重新登入
   Then 系統保留上次主題偏好並自動套用。
